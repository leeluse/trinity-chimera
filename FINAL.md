# Trinity AI Trading System: Final Technical Documentation (Phases 1-5)

This document serves as the definitive record of the transition from a mock-based backend to a fully autonomous, self-evolving AI trading system.

---

## 📌 Project Vision
The Trinity AI Trading System is an ecosystem where LLM agents autonomously generate trading strategies, validate them through rigorous backtesting, and evolve their code in a closed loop, simulating a biological evolutionary process applied to quantitative trading.

### 🎯 The North Star Metric: Trinity Score
The system optimizes for the **Trinity Score**, a composite KPI designed to balance raw profit with risk-adjusted stability:
$$\text{Trinity Score} = (\text{Return} \times 0.40) + (\text{Sharpe} \times 25 \times 0.35) + ((1 + \text{MDD}) \times 100 \times 0.25)$$

---

## 🛠️ Phase 1: Supabase Infrastructure & Data Layer
**Goal:** Establish a persistent source of truth for agents and their evolutionary lineage.

### Data Model (PostgreSQL)
- **`agents`**: Stores agent personas (e.g., Momentum Hunter, Mean Reverter) and references the currently active strategy.
- **`strategies`**: Versioned storage of LLM-generated Python code, parameters, and the rationale behind changes.
- **`backtest_results`**: Historical performance metrics (Trinity Score, Return, Sharpe, MDD) per strategy version.
- **`improvement_logs`**: Detailed records of evolution steps, including LLM analysis of what was improved.

### Core Integration
- Implemented `SupabaseManager` for asynchronous CRUD operations, enabling seamless versioning of strategies and tracking of performance over time.

---

## 🧪 Phase 2: Dynamic Strategy Sandbox (B-Mode)
**Goal:** Safely execute arbitrary Python code generated by LLMs without compromising system security.

### Security & Isolation Architecture
- **AST-based Validation**: The `StrategyLoader` uses Python's `ast` module to block forbidden imports (`os`, `sys`, `subprocess`, `shutil`, `socket`, `pickle`, `threading`, `multiprocessing`) and dangerous functions (`open`, `eval`, `exec`, `__import__`).
- **Execution Isolation**: Strategies are executed within a separate `multiprocessing.Process` to ensure that a crash in the strategy does not bring down the orchestrator.
- **Hard Timeouts**: A 30-second hard timeout is enforced via process termination to prevent infinite loops.
- **Polymorphism**: All strategies must inherit from `StrategyInterface`, ensuring a consistent API (`generate_signal()`).

---

## 📈 Phase 3: Robust Backtesting Engine
**Goal:** Eliminate "optimism bias" and overfitting to ensure strategies survive in real market conditions.

### Anti-Overfitting Guardrails
- **IS/OOS Splitting**: Data is divided into In-Sample (Training) and Out-of-Sample (Validation) sets (typically 30 days each).
- **Validation Gate**: A strategy is only accepted if its OOS Trinity Score is $\ge 70\%$ of its IS score. This prevents the system from promoting strategies that merely "memorized" the training data.

### Conservative Cost Model
- **Real-world Friction**: Integrated a penalty system involving a fixed fee (0.05%) and random slippage (0.01% to 0.03%) per trade.
- **Impact**: Ensures that "paper profits" are stripped away, leaving only strategies with a genuine alpha that can withstand trading costs.

---

## 🧠 Phase 4: Autonomous Evolution Orchestrator
**Goal:** Build the "brain" that drives the self-improvement loop.

### The Evolution Loop (State Machine)
`TRIGGERED` $\rightarrow$ `GENERATING` $\rightarrow$ `VALIDATING` $\rightarrow$ `COMMITTING`

### Key Innovations
- **Adaptive Trigger System**: Evolution is triggered by four layers:
    - **L1 (Regime-Shift)**: Immediate response to market regime changes (HMM-based).
    - **L2 (Performance Decay)**: Urgent evolution when Trinity Score $\le 80\%$ of historical average.
    - **L3 (Competitive Pressure)**: Motivation to evolve when trailing the top agent's score significantly.
    - **L4 (Heartbeat)**: Routine tuning every 14 days.
- **C-mode Context Feedback**: LLM is provided with a high-density prompt including:
    - Current code, Trinity Score, MDD-heavy loss logs, Market Regime, S-curve history, and the top agent's strategy characteristics.
- **Self-Correction Loop**: If generated code fails validation or syntax checks, the orchestrator feeds the error traceback back to the LLM for up to 3 correction attempts.

## 🌐 Phase 5: API & Frontend Integration (Final)
**Goal:** Transform the "black box" evolution into a transparent, real-time observable system.

### Real-time Observability
- **Supabase Realtime**: Frontend subscriptions ensure that as soon as an agent evolves and commits a new strategy, the dashboard updates instantly without refresh.
- **Strategy Code Viewer**: A dedicated UI component to inspect the actual Python code generated by the LLM, allowing humans to audit the "thinking" of the AI.

### End-to-End Flow
`Market Change` $\rightarrow$ `Adaptive Trigger` $\rightarrow$ `LLM Generation (C-mode)` $\rightarrow$ `Self-Correction` $\rightarrow$ `IS/OOS Validation` $\rightarrow$ `Supabase Commit` $\rightarrow$ `Real-time UI Update`.
